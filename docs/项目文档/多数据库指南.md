# 多数据库实体存放与访问方案

> 本文档为**设计方案**，描述项目接入多数据库时实体类的存放规范与 SqlSugar 多库访问方案。
> 当前项目已配置 `MainDb`（主库 `LogisticsProduction_DB`）与 `91Db`（`cwbase0006`）两个连接，但实体存放与访问层尚未按多库规范组织。本文档给出目标结构与落地步骤。

---

## 一、现状与问题

### 1.1 现状

| 项 | 现状 | 位置 |
|---|---|---|
| 连接配置 | 已配置 `MainDb` + `91Db` 两个连接字符串 | [appsettings.json](../../appsettings.json) |
| ORM | SqlSugarCore 5.1.4.160 | [LogisticsProduction.Net8.csproj](../../LogisticsProduction.Net8.csproj) |
| 上下文工厂 | `CreateClient(connectionName)` 按连接名创建单连接 `SqlSugarClient`，`DbType` 硬编码 `SqlServer` | [Infrastructure/Persistence/DbContextFactory.cs](../../Infrastructure/Persistence/DbContextFactory.cs) |
| 仓储基类 | 构造时 `CreateClient()`（默认 MainDb），每个仓储绑定单个 client | [Infrastructure/Persistence/BaseRepository.cs](../../Infrastructure/Persistence/BaseRepository.cs) |
| 实体目录 | 主库与 91 库实体混放在 `Domain/Entities/` 同一命名空间 `...Domain.Entities` | [Domain/Entities/](../../Domain/Entities/) |

### 1.2 问题

1. **实体归属不清**：`LogisticsContainer`（主库）与 `SCLZQD2_BARCODE`、`PritMod`（91 库）位于同一目录、同一命名空间，无法从代码看出归属哪个库。
2. **91 库实体原未编译**：`PritMod`、`SCLZQD2_BARCODE` 原被 csproj `<Compile Remove>` 排除（归档于 [_archive/code/](_archive/code/)），现已迁入 [Domain/Entities/Db91/](../../Domain/Entities/Db91/) 并启用编译（详见下文「实施进度」）。
3. **多库无法自动路由**：`BaseRepository` 默认只创建 MainDb 的 client，访问 91 库需另行处理，且无实体级归属标识。
4. **数据库类型硬编码**：`DbType.SqlServer` 写死，未来扩展 MySQL/Oracle 等需改源码。
5. **重名风险**：两个库若有同名表，同命名空间下会冲突。

---

## 二、目标

- 实体类**按库分目录、分子命名空间**存放，归属一目了然。
- 通过 SqlSugar **`[Tenant]` 特性 + `SqlSugarScope` 多连接容器**实现查询自动路由到正确数据库。
- `DbContextFactory` 升级为多库容器，`DbType` 参数化以支持未来多类型。
- `EntityGenerator` 按库分别输出到对应子目录。
- 不破坏现有主库业务代码（渐进迁移）。

## 三、实体存放方案

### 3.1 目录与命名空间结构

```
Domain/Entities/
├── MainDb/                      # 主库 LogisticsProduction_DB
│   ├── BaseEntity.cs            # namespace LogisticsProduction.Net8.Domain.Entities.MainDb
│   └── LogisticsContainer.cs
└── Db91/                        # 91 库 cwbase0006
    ├── SCLZQD2_BARCODE.cs       # namespace LogisticsProduction.Net8.Domain.Entities.Db91
    └── PritMod.cs
```

### 3.2 关于 91 库命名空间命名（重要）

> **`91Db` 不能直接用作 C# 命名空间标识符**——C# 标识符不能以数字开头。

可选命名（按需选择，保持与连接名映射清晰即可）：

| 子目录/命名空间 | 连接名 | 说明 |
|---|---|---|
| `Db91` | `91Db` | 简洁，推荐 |
| `System91` | `91Db` | 语义更明确 |
| `<业务系统名>` | `91Db` | 如 91 库是某 ERP/MES，可用其系统名 |

**建议统一用 `Db91`**，并在方案落地时同步调整连接名映射约定。

### 3.3 实体归属特性

每个实体用 SqlSugar `[Tenant]` 特性声明归属的 `ConfigId`，查询时自动路由：

```csharp
// 91 库实体
using SqlSugar;
namespace LogisticsProduction.Net8.Domain.Entities.Db91;

[SugarTable("SCLZQD2_BARCODE")]
[Tenant("91Db")]                 // 绑定到 ConfigId="91Db" 的连接
public partial class SCLZQD2_BARCODE { ... }

// 主库实体
using SqlSugar;
namespace LogisticsProduction.Net8.Domain.Entities.MainDb;

[SugarTable("LogisticsContainer")]
[Tenant("MainDb")]               // 绑定到 ConfigId="MainDb" 的连接
public class LogisticsContainer : BaseEntity { ... }
```

## 四、SqlSugar 多库访问方案

### 4.1 改造 DbContextFactory

将单连接 `SqlSugarClient` 升级为多连接容器 `SqlSugarScope`，注册所有库的 `ConnectionConfig`：

```csharp
// Infrastructure/Persistence/DbContextFactory.cs（改造后示意）
public class DbContextFactory
{
    private readonly AppConfigService _configService;
    private readonly SqlSugarScope _scope;

    public DbContextFactory(AppConfigService configService)
    {
        _configService = configService;

        _scope = new SqlSugarScope(new List<ConnectionConfig>
        {
            new()
            {
                ConfigId        = "MainDb",
                ConnectionString = _configService.GetConnectionString("MainDb"),
                DbType          = DbType.SqlServer,
                IsAutoCloseConnection = true,
                InitKeyType     = InitKeyType.Attribute
            },
            new()
            {
                ConfigId        = "91Db",
                ConnectionString = _configService.GetConnectionString("91Db"),
                DbType          = DbType.SqlServer,
                IsAutoCloseConnection = true,
                InitKeyType     = InitKeyType.Attribute
            }
        });
    }

    /// 默认返回主库 scope（实体凭 [Tenant] 自动路由到对应库）
    public ISqlSugarClient GetClient() => _scope;

    /// 显式获取指定库的 client（兼容旧用法）
    public ISqlSugarClient GetClient(string configId) => _scope.GetConnectionScope(configId);
}
```

> `SqlSugarScope` 是单例容器，多个 `ConnectionConfig` 以 `ConfigId` 索引。实体标注 `[Tenant("ConfigId")]` 后，`Queryable<T>()` 会自动切换到对应连接。

### 4.2 改造 BaseRepository

```csharp
// Infrastructure/Persistence/BaseRepository.cs（改造后示意）
public abstract class BaseRepository<T> where T : class, new()
{
    protected readonly ISqlSugarClient Db;

    protected BaseRepository(DbContextFactory dbFactory)
    {
        // 不再传 connectionName——实体凭 [Tenant] 自动路由
        Db = dbFactory.GetClient();
    }

    // 其余 CRUD / 存储过程 / 事务方法保持不变
}
```

- **跨库事务**：使用 `_scope.BeginTran()`（配合 `AsTenant()`），可同时提交/回滚多个库。
- **指定库操作**：需要显式指定库时，注入 `DbContextFactory` 并调用 `GetClient("91Db")`。

### 4.3 单库仓储 vs 跨库仓储

- **单库仓储**（推荐，绝大多数场景）：实体已用 `[Tenant]` 标注，仓储无需关心连哪个库。
- **跨库联表**：SqlSugar 跨库联表需用 `// xxx` 库前缀语法，复杂场景建议拆成各自查询后在应用层组装。

## 五、EntityGenerator 改造方案

当前 [EntityGenerator.cs](../../Tools/EntityGenerator.cs) 固定输出到 `Domain/Entities`、命名空间 `...Domain.Entities`。改造为按库输出：

```csharp
// 改造要点
public EntityGenerator(string connectionString, string outputPath, string namespaceName)
{
    // outputPath   → Domain/Entities/MainDb 或 Domain/Entities/Db91
    // namespaceName→ ...Domain.Entities.MainDb 或 ...Domain.Entities.Db91
}
```

- 生成时通过模板追加 `[Tenant("<configId>")]` 特性。
- `CreateClassFile` 输出目录与命名空间均按库区分。
- [Controllers/Dev/EntityGeneratorController.cs](../../Controllers/Dev/EntityGeneratorController.cs) 的入参增加 `database`/`outputPath` 字段，指定目标库与子目录。

## 七、注意事项

- **命名空间合法性**：91 库子目录不可用 `91Db`（数字开头非法），统一用 `Db91`。
- **`[Tenant]` 与 `ConfigId` 必须一致**：特性值与 `ConnectionConfig.ConfigId` 严格匹配，否则路由失败。
- **BaseEntity 归属**：`BaseEntity` 仅主库审计字段使用，放 `MainDb/`；若 91 库也有审计字段，单独定义 91 库的基类或用独立实体。
- **DbType 参数化**：若未来接入非 SQL Server 库，将 `DbType` 从配置读取（如 appsettings 增加 `DbType` 字段）。
- **渐进迁移**：可先迁主库实体并验证，再迁 91 库，避免一次性大改。
